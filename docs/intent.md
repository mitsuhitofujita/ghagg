### 開発エージェントへのコンテキスト・ポリシー指示書

**1. プロジェクトの真の目的（Why）**
* **活発なレビュー文化の可視化:** 当開発チームの強みである「厚いレビュー」と「エンジニア間の活発な議論」の実態を把握し、レビュー負荷の偏りの発見や、チームのふりかえり（プロセス改善）のための健康診断データとして活用する。
* **「グッドハートの法則」の回避:** 集計したPR数、コメント数、アプローブ数は、メンバー個人の評価指標や目標としては**一切使用しない**。指標をゲーム化（意味のないPRやコメントの量産）させないことを最優先とする。

**2. システム設計の基本ポリシー（Architecture Philosophy）**
* **KISS & YAGNIの徹底:** 評価指標として使わないため、厳密なリアルタイム性やデータベースによる状態管理は不要。過剰な作り込みを避け、「ローカルへのJSONファイルダンプ」と「メモリ上での集計」という、最もシンプルで保守運用コストの低いスナップショット・アプローチを採用する。
* **N+1問題の回避:** GitHub APIのレートリミット枯渇を防ぐため、REST APIではなく GraphQL API を採用し、必要なネストされたデータを効率的に一括取得する。

**3. データ抽出におけるドメイン要件（Domain Requirements）**
* **インラインコメントの重視:** PR全体へのコメントだけでなく、コードの特定の行に対するコメント（レビューコメント）が非常に多いチーム文化である。そのため、単一PRに対して数百件規模のコメントやレビューが発生し得ることを前提とし、それらを深く掘り下げて捕捉できる取得戦略をとること。
* **集計の軸:** 指定期間内における、メンバー単位での「PRオープン数」「アプローブ数」「実態に即したコメント総数」を算出すること。

**4. エージェントへの期待（Expectation）**
上記の「チームの文化（厚いレビュー）」と「設計の割り切り（シンプルさの追求）」のコンテキストを深く理解した上で、複雑さを排した見通しの良いデータ取得・集計スクリプトの実装を提案・実行してください。
